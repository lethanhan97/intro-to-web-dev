Web Dev Notes

Classes and IDs

General Info

Classes are only used by you! If you use a bad name, it will only mess you!

IDs are supposed to be unique while classes can be repeated. IDs are like sledgehammers. Most problems today use normal hammers. But when there's a problem that requires a sledgehammer, then use it

Classes aim to be as generic as possible, while IDs aim to be as unique as possible

HTML is case insensitive. Hence naming is always in kebab cases this-is-kebab-case

Dont screw up screen readers

Naming

With classes, name it semantically. Name what it does, not how it looks like

Meta HTML

lang="en" tells the browser what the language of the document is gonna be

  • script tag is for importing JS
  • style is for inline styling
  • link is for importing CSS

CSS

General Info

List of rules to tell the browser how to display a current element.

em is measured by the length of the m letter in a font size

Selectors of parents will affect its childrens

Always style on classes. Don't style on tags unless you need to

The cascade

Avoid it. But if you have to, then read on...

Specificity wins. .class1 .class2 {} will win against .class2 {}

Rules that are defined later win

Classes are more specific than tags IDs can overrite tags and classes (dont use ID)

!important is even more specific. Dont use it

Pseudoclasses

describes the state of a class e.g: hover, active, focused, first-child, nth-child()

have the same specificity of a class

Pseudoelements are elements generated by CSS, not HTML

  • Most browsers currently support Flex. Not all browsers support Grid (if u use grid ure cutting 15-20% of your users)
